# Phase 8: STM32MP1 (CP700P) RFID 카드리더기 하드웨어 통합 보고서

## 1. 개요
본 문서는 실제 타겟 보드(STM32MP15-eval) 환경에서 시리얼 기반(UART) RFID 모듈을 통해 결제/인증 카드(IdToken)를 스캔하고, 이를 OCPP 2.0.1 클라이언트 비동기 데몬(`main.py`)에 안전하게 연동한 과정과 기술적 해결책을 다룹니다.

## 2. 하드웨어 포트 검증 및 매핑
초기 `ttySTM1` 포트 가설을 폐기하고, SSH를 통한 커널 레벨 진단(dmesg 및 Linux tty driver bytes 조회)을 통해 물리적 결선 상태를 입증했습니다.

- **추적 기법**: `cat /proc/tty/driver/stm32-usart` 명령을 통해 시스템 부팅 이후 물리적인 수신(RX) 바이트가 존재하는 포트를 색출.
- **결과**: `ttySTM6` 포트에서만 물리 수신 데이터(RX=39)가 포착되어 카드리더기 결선 포트로 최종 확정.
- **최종 매핑**: `/dev/ttySTM6` (Baudrate: 9600)

## 3. 원시 바이너리 프레임(Raw Binary) 파싱 도입
일반적인 ASCII 시리얼 출력(ex: "12345678\n")을 기대한 초기 파싱 로직에서, UTF-8 디코딩 시 특수문자(`CpS` 등)로 문자가 깨지는 현상이 발생했습니다.

### 3.1 현상 분석
리더기는 문자열이 아닌 **Wiegand/STX 기반의 순수 16진수 바이너리 프레임**을 UART 채널로 전송하고 있었습니다.
- 원본 들어온 바이트(Hex): `02 00 09 43 10 40 00 99 70 14 89 53 3D` (총 13바이트)

### 3.2 Slicing 추출 알고리즘 적용
디코딩을 완전히 배제하고 파이썬 `bytes.hex()`를 이용해 스트림을 16진수 대문자 문자열로 다루도록 변환했습니다.
- 프레임 헤더(`02`: STX)로 시작하고 길이가 충분할 경우, 정확히 카드 번호 데이터 구역(8번째 문자부터 24번째 문자)을 Slicing(`raw_hex[8:24]`)
- 추출된 고유 토큰: `1040009970148953`

이 정제된 토큰은 즉시 OCPP `AuthorizeRequest`의 `IdToken` 필드로 삽입되어 서버(CSMS)로 전송됩니다.

## 4. 백그라운드 비동기 데몬 최적화
RFID의 수신 대기(`ser.read(ser.in_waiting)`)는 Thread 블로킹을 유발하므로 메인 OCPP 웹소켓 Heartbeat 루프를 멈추게 할 위험이 있습니다.
- **해결책**: `asyncio.to_thread`를 사용하여 시리얼 포트 읽기 전용 래퍼(`blocking_read_rfid`)를 별도의 OS 쓰레드로 위임.
- **결과**: 단 0.1초 수준의 메인 루프 간섭 없이도, 카드가 닿는 즉시 비동기로 콜백 신호를 발송하여 OCPP `handle_rfid_scan()` 루틴을 실행하게 됩니다.

## 5. 최종 시연 흐름
1. `sudo venv/bin/python3 src/main.py` 실행
2. 메인 데몬: `ttySTM6` 9600 Baudrate로 백그라운드 리스닝 시작
3. 사용자 카드 접근 시 13바이트 물리 패킷 유입 -> `02000943...3D`
4. 데몬 파싱: `1040009970148953` (고유값 추출)
5. `OCPP AuthorizeRequest` 발송 -> `[2, "<UUID>", "Authorize", {"idToken": {"idToken": "1040...", "type": "ISO14443"}}]`
6. 통신 검증: 콘솔에 전송 로그(`WS SEND`) 표출 확인 완료
